## Skill: Bitrix AI Challenge — модуль «Избранное»

### Когда применять

Открывай этот файл, если задача связана с реализацией/доработкой тестового модуля «Избранное (Wishlist)» из `bitrix_ai_challenge.md`.

### Источник истины

- `bitrix_ai_challenge.md` — первичные требования, ограничения и ожидаемая структура.
- `pages/` — справочник по актуальному D7 API и паттернам.

### Жёсткие ограничения (нельзя нарушать)

- Не модифицировать ядро (`/bitrix/*`).
- Не использовать глобальные обработчики в `init.php`.
- Не делать прямых SQL‑запросов (только ORM).
- Не использовать старое ядро (`CIBlock*`, `CUser` и т.п.).
- Совместимость: Bitrix 23.x+ и PHP 8.2+.

### Обязательный функционал

#### Хранилище

- **Авторизованный пользователь**: собственная таблица через ORM `DataManager`.
- **Гость**: cookie, **шифрованные** (`CryptoCookie` по ТЗ).

#### Миграция «гость → пользователь»

- При авторизации переносить избранное из cookie в БД.
- Событие по ТЗ: `OnAfterUserAuthorize`.
- Дубликаты не создавать.

#### REST API (контроллер)

ТЗ требует API через `\Bitrix\Main\Engine\Controller`:

- `POST favorites.add`
- `POST favorites.remove`
- `GET favorites.list`
- `GET favorites.getProducts`

Практика:

- Действия контроллера — публичные методы с суффиксом `Action` (например, `addAction`, `removeAction`).
- Обязательны: проверка HTTP‑метода, CSRF, аутентификация/гость‑логика (см. `pages/framework/controllers.md`).

#### Публичный компонент `vendor:favorites.button`

- Кнопка add/remove + визуальное состояние.
- AJAX без перезагрузки.
- Современный минималистичный UI, анимации, адаптивность; тёмная/светлая тема опционально.
- Параметры:
  - `PRODUCT_ID`
  - `SHOW_COUNTER`
  - `BUTTON_SIZE` (`small|medium|large`)

Подсказка по реализации AJAX у компонентов: `pages/framework/components.md` (Controllerable, signed params, `BX.ajax.runComponentAction`).

#### Кеширование и инвалидция

- Список избранного для авторизованного пользователя должен кешироваться.
- Инвалидировать при изменении/удалении товара инфоблока через **теги** (тегированный кеш D7).
- Ожидаемые события по ТЗ:
  - `OnAfterIBlockElementDelete` — удалить товар из избранного всех пользователей
  - `OnAfterIBlockElementUpdate` — инвалидировать кеш при изменении товара

См. также: `pages/performance/caching.md`.

#### Админка

Страница настроек модуля:

- выбрать инфоблок каталога
- TTL cookie для гостей
- включить/выключить функционал модуля

### Нефункциональные требования

- API‑запросы \(цель\) ≤ 100ms: избегай лишних запросов, используй кеш и правильные индексы.
- Безопасность: валидация всех входящих параметров, проверка существования товара, CSRF для API‑методов, шифрование cookie.

### Рекомендуемые “проверки на выходе”

- Нет обращения к `CIBlock*`/`CUser`/прямому SQL.
- Все модифицирующие действия — не через GET, с CSRF.
- Данные, попадающие в HTML/JS, экранируются (см. `pages/security/xss.md`).
- Кеши валидно сбрасываются по тегам и не “протекают” между сайтами/контекстами.
